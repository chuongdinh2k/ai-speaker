# S3 Audio Storage with Pre-signed URLs

**Date:** 2026-04-12  
**Status:** Approved

## Problem

Audio files generated by TTS are currently saved to local disk and served via FastAPI's `StaticFiles` mount at `/audio/<filename>`. This endpoint is fully public — any URL can be played without authentication. The goal is to store audio on S3 and ensure only authenticated users can access it.

## Chosen Approach

**Pre-signed URL generated at upload time (Option A)**

When `synthesize_speech` uploads audio to S3, it immediately generates a pre-signed URL with a 1-hour expiry and returns it. The chat API returns this URL in the `audio_url` field. The frontend fetches audio using the JS Fetch API with the pre-signed URL — no auth header needed since the signature is embedded in the URL itself. Security comes from the fact that only the backend (with AWS credentials) can generate valid pre-signed URLs, and they expire after 1 hour.

## Components Changed

### `app/config.py`
Add new settings:
- `aws_access_key_id: str`
- `aws_secret_access_key: str`
- `s3_bucket_name: str`
- `s3_region: str`
- `s3_presigned_url_expiry: int = 3600`

### `app/voice/service.py`
Replace local file write in `synthesize_speech` with:
1. Upload MP3 bytes to S3 via `boto3`
2. Generate pre-signed URL immediately after upload
3. Return the pre-signed URL instead of `/audio/<filename>`

The `transcribe_audio` function is unchanged.

### `app/main.py`
Remove the `StaticFiles` mount for `/audio` — no longer needed.

### `.env`
Add:
```
AWS_ACCESS_KEY_ID=...
AWS_SECRET_ACCESS_KEY=...
S3_BUCKET_NAME=...
S3_REGION=...
```

## Data Flow

```
Chat API → synthesize_speech()
              → OpenAI TTS → MP3 bytes
              → boto3 upload to S3 (same region as EC2, ~50-100ms)
              → generate pre-signed URL (1hr, local crypto ~1ms)
              → return presigned URL
         → audio_url = "https://<bucket>.s3.<region>.amazonaws.com/..."
         → frontend plays audio via JS Fetch API
```

## Error Handling

If S3 upload fails, `synthesize_speech` raises an exception. The existing try/except in `chat/service.py:handle_chat_message` already catches TTS failures and returns `audio_url: null` gracefully — no change needed.

## S3 Bucket Configuration

The S3 bucket should have:
- **Block all public access** enabled (objects are private by default)
- Bucket in the **same AWS region** as EC2 to minimize upload latency

## Dependencies

- Add `boto3` to `requirements.txt`
